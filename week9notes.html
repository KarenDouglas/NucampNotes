Intro to Redux

About MVC Design Pattern

  • aka MVC FrameWork
  • a software design pattern is not code, or a library
  • typically used to develop presentation layer/client-side
  • a key concept for MVC is its separation of concerns
  • it has a bidirectional dataflow
  • its presentation layer is divided into three parts: Model, View, Controller
  • becomes unpredictable at scale, difficult to add new features without causing unexpected cascading effect in other parts of the code

MVC Design Pattern

a softward design pattern that is a reusable approach for how to structure an application

Separation of Concerns

facilitates independent development, testing, and maintenance of different parts of an app

MVC Model

  • handles app state and logic
  • stores the main app logic and data for the frontend
  • responds to queries about state(from View/Controller) form the frontend
  • responds to requests to update state (from Controller)
  • it may also notify View/Controller when its state changes

MVC View

  • presents info to user
  • renders Model into a form suitable for display, typically a UI element
  • can query Model directly about its state and update itself based on changes

MVC Controller

  • mediates between model and view
  • recieves and processes info recieved from the View(from user interactions)
  • can instruct both Model and View to update

The Flux Architecture

About Flux

  • designed to scale better than MVC pattern Design for complex and large applications
  • it has a strictly defined unidirectional data flow
  • it separates it concerns into: Action,Dispatcher,Store and View
  • it doesn't provie actual code

Flux

an approach for how to structure your code on the frontend,considered an evolution of MVC Design Pattern

Flux Flow of Concerns

  1. Action to Dispatcher:
    when an action of some kind goes into the system it goes to the dispatcher(there is one cental dispatcher that acts as a traffic controller and routes actions to stores)
  2. Dispatcher to Stores:
    the stores hold app state and logic,this is similar to MVC Models
  3. Stores to View:
    the view updates when stores are changed
  4. View to Action:
    another action can enter the system from View. the dispatcher makes sure that actions are processed one at a time to prevent cascading effects

Overview: Redux

Redux

  • considered to be an evolution of Flux
  • also inspired by the Elm functional programming libraries
  • its considered a predictable state container for JS apps
  • it provides code that stores application state and a consistent wa to access and update that state from anywhere in the app

Redux in Theory

  • Principles of Redux: Single Source of Truth, State is Read-Only, Changes are made with pure functions
  • Single Source of Truth: unlike in flux, Redux has only one store which holds a single state object tree that contains the whole state of your entire app as objects within that tree
  • State is Read-Only: the only way to change state is with an action - an object that describes what happened
  • Changes are Made with Pure Functions: pure functions are functions that have no side effects and given the same inputs,always returns the same outputs

Object Tree

Tree-like data structure; each node contains an object; DOM is an example of this

reducers

what redux calls pure functions, it takes the previous state and an action and returns the next state

pure functions

functions that have no side effects and given the same inputs,always returns the same outputs(i.e. Map.floor()

immutability

a programming concept in which data does not get mutated, only replaced with new data

Redux in Practice

  • Redux Flow of Concerns: Actions---Reducers--Store--View--Actions
  • as with Flux, there is only one data-flow

createStore( )

a pure function that creates the Redux store which holds the state(as an object tree)

connect( )

a pure function that generates container components that wraps arounds other components to subscribe them to the store

mapStateToProps( )

a pure function that is passed as callback to connect( ) function. Its called whenever state changes.. It recieves entire state tree and returns an object that contains only the data needed by the component
